GRU'S MINION SWARM

A Claude-powered multi-agent swarm, themed as the Minions under Imperator Gru, running solar-system-evolution-ruleset.md live — with a playable Asteroids-style side mode when you'd rather fly than watch.
PROJECT SUMMARY — 2026-07-23

01What this is

Eight Claude agents, one per planet, each independently deciding how to survive its own hazard forecast every epoch — under fog of war (they're told the hazard type, never its severity). A ninth agent, Gru, coordinates the fleet: every epoch he delegates to all eight living garrisons in a single message, and all eight decisions run as genuinely concurrent Claude calls. The theme (Minions, X-Wings, Gru's flagship, a playable dodge-the-death-ray minigame) sits on top of a real multi-agent system — not the other way around.

The engine (engine.js) is deterministic and owns every hazard roll and resolution formula; the agents (agents.js) never see a rolled severity, only the forecasted type — the ruleset's "fog of war" premise, enforced in code, not just in the prompt.

03The parallelism story

Gru's system prompt instructs him to delegate to every living garrison in one turn — one tool_use block per planet, forced with tool_choice:"any" so he can't just write prose instead. The server then executes every dispatched specialist call with Promise.all, not a loop with await inside. Measured on a real run:

Gru → 8 garrisons: all 8 tool_use blocks arrive in a single coordinator response — 0-1ms spread between the first and last specialist_start event.
8 specialists → Gru: wall-clock for all 8 parallel Haiku calls was ~1.9s in one measured run; the sum of their individual latencies was ~13s — an ~6-7x speedup from real concurrency, not an illusion of it.

This is checked by a standing regression script, not just eyeballed once: npm run test:fanout asserts the start-spread and the wall-clock-vs-serial-sum ratio on every run.

04Mechanics: cooperate and compete

Each Minion sees the full rival roster (population and CAI rank) and Gru's shared Dominion meter (the fleet's combined population) every epoch, and is explicitly framed with a dual objective: climb the individual leaderboard, but don't be the reason the fleet's shared score collapses. They can also spend 3 AP to raid a named rival — offense ((STR+MOB)/2) versus the target's defense (STR) decides how much population actually transfers.

This produces genuine emergent behavior, not scripted outcomes. In testing, Mercury and Venus independently raided each other in the same epoch — resolved correctly because raids snapshot pre-epoch stats before any transfer applies, so simultaneous raids don't depend on iteration order. In another run, Saturn and Uranus both raided Earth independently in the same turn.

On the visual side: the Dominion meter isn't decorative — a real swing in it (growth or a setback) shakes the scene, and the fleet's on-screen formation radius eases outward as Dominion climbs toward its target, so the galaxy visibly expands as the conquest grows.

05Safety & verification

engine.sanitizeAction() is the one place untrusted model output meets the deterministic engine — it never trusts the LLM's arithmetic. A regression suite (npm run eval) gates every change to engine.js or agents.js:

  • 11/11 checks passing — 8 capability, 3 regression
  • CRITICAL-severity: AP overspend is clamped, negative-invest can't manufacture free AP, raids can't target self or an unknown planet
  • MEDIUM-severity: null/non-numeric/garbage action payloads never throw
A real bug, found and fixed: clicking RESET while an epoch was mid-flight swapped in a brand-new engine instance; the in-flight epoch then tried to resolve against it — a fresh engine that never had beginEpoch() called on it, so the crash was Cannot read properties of undefined (reading 'get'). Fixed with a generation counter bumped on reset, checked before every broadcast — a stale epoch's events (including in-flight specialist_done events already en route) are now silently discarded instead of corrupting state. The fix shipped with both a permanent regression test and a reproduction script (npm run test:reset-race) that fires the exact race on demand.

Pricing: Haiku 4.5 is $1.00 / $5.00 per million input/output tokens. Extrapolated, a full 100-epoch "Horizon run" (per the ruleset's own end condition) costs on the order of $2.30 in inference — cheap enough to run the whole simulation many times over while iterating.

Cache read/write is 0 tokens, on purpose (verified, not a bug): the shared specialist system prompt is marked cache_control: ephemeral because it's byte-identical across all 8 calls and every epoch — but Haiku 4.5's minimum cacheable prefix is 4,096 tokens, and that prompt is only ~380 tokens. Below the minimum, the cache marker is a documented no-op: no error, just cache_creation_input_tokens: 0 forever. A bigger shared context (e.g. the full ruleset text embedded verbatim) would cross the threshold and unlock the real ~90% cost reduction prompt caching is capable of.

07Fun Mode

Type G, R, U in sequence anywhere on the page to scramble into an Asteroids-style dodge game: steer with arrows/WASD, fire with Space, dodge Gru's telegraphed wrath-ray sweeps, shoot asteroids for score. The main simulation keeps running server-side the entire time you're away — flipping back to Command view shows wherever the epoch loop got to.

09Honest limitations

  • Every visual claim in this project (Minion art, Gru's portrait compositing, screen shake, drag-and-drop feel) was verified by a human, not by the agent that built it — there's no browser tool in this build loop.
  • Prompt caching is configured correctly but currently inert (see §06) — it's a real lever, not yet a real saving, at this prompt size.
  • Fixed roster of 8 planets, matching the ruleset's suggested set — moons/dwarf planets are a documented expansion path in the ruleset, not implemented here.
  • Speed control (1×/2×/4×) only shortens the pause between epochs — Claude API latency for the coordinator + parallel specialists (~2-6s/epoch) is the real floor either way.
